# KMS — Interview Questions

---

## Easy

### 1. What is AWS KMS and what problem does it solve?

**Answer:**
AWS Key Management Service (KMS) is a managed service that makes it easy to create and control cryptographic keys used to encrypt and decrypt data. It solves the problem of secure key management — without KMS, teams would need to build, operate, and secure their own key management infrastructure, which is complex and error-prone. KMS integrates natively with over 100 AWS services, allowing encryption of data at rest and in transit without requiring developers to manage the underlying cryptographic complexity.

---

### 2. What is a Customer Master Key (CMK) / KMS Key?

**Answer:**
A KMS Key (formerly called a Customer Master Key or CMK) is the primary resource in AWS KMS. It is a logical representation of a cryptographic key that never leaves KMS unencrypted. There are three types:
- **AWS Managed Keys**: Created and managed by AWS on your behalf for a specific service (e.g., `aws/s3`). You cannot manage rotation or key policies directly.
- **Customer Managed Keys**: Created and managed by you. You control the key policy, rotation, and deletion.
- **AWS Owned Keys**: Owned and managed entirely by AWS, used across multiple accounts. You have no visibility or control.

---

### 3. What is the difference between symmetric and asymmetric KMS keys?

**Answer:**
- **Symmetric keys**: Use a single 256-bit AES-GCM key for both encryption and decryption. The key never leaves KMS. This is the most common type and is used for most AWS service integrations.
- **Asymmetric keys**: Consist of a public/private key pair. The public key can be downloaded and used outside AWS. The private key never leaves KMS. Asymmetric keys support RSA (for encryption or signing) and ECC (for signing). They are used for digital signatures, public key encryption, and scenarios where the encryptor and decryptor are in different trust boundaries.

---

### 4. What is a Data Encryption Key (DEK) and how does envelope encryption work?

**Answer:**
A Data Encryption Key (DEK) is a symmetric key generated by KMS used to encrypt actual data. Envelope encryption is the process where:
1. KMS generates a plaintext DEK and an encrypted copy of the DEK (encrypted under a KMS key).
2. The plaintext DEK encrypts the data.
3. The plaintext DEK is immediately discarded from memory.
4. Only the encrypted DEK is stored alongside the ciphertext.

To decrypt, you send the encrypted DEK to KMS, which decrypts it using the KMS key, returning the plaintext DEK to decrypt the data. This approach means KMS only handles small key material, not large datasets, improving performance and reducing exposure.

---

### 5. What is key rotation in KMS and how does it work?

**Answer:**
Key rotation is the process of periodically generating new cryptographic key material for a KMS key. For customer managed symmetric keys:
- **Automatic rotation**: Enabled by you; AWS rotates the key material every year (365 days). The key ID, ARN, and key policies remain unchanged.
- **On-demand rotation**: You can trigger rotation manually at any time (in addition to annual automatic rotation).
- Old key material is retained indefinitely so data encrypted under previous versions can still be decrypted.
- AWS Managed Keys are automatically rotated every year by AWS.
- Asymmetric keys and HMAC keys **do not support automatic rotation**.

## Medium

### 1. Explain KMS key policies and how they differ from IAM policies for controlling access to KMS keys.

**Answer:**
KMS key policies are resource-based policies attached directly to a KMS key. They are the **primary access control mechanism** for KMS keys. Unlike most AWS resources where IAM policies alone can grant access, KMS requires the key policy to explicitly allow access.

**Key differences:**

| Aspect | Key Policy | IAM Policy |
|---|---|---|
| Attachment | Attached to the KMS key | Attached to an IAM identity |
| Required | Yes — key policy must allow access | Supplementary |
| Scope | Controls who can use/manage the key | Controls what the identity can do |
| Cross-account | Enables cross-account access | Must also be present in target account |

**How they interact:**
- For same-account access: The key policy must allow the account root or specific principals. If the key policy grants access to the root account (`arn:aws:iam::ACCOUNT_ID:root`), IAM policies can then further delegate access to IAM users/roles.
- For cross-account access: The key policy must explicitly allow the external account's principal, AND the IAM policy in the external account must also allow the KMS actions.

**Best practice**: Always explicitly define key administrators and key users in the key policy. Avoid overly permissive policies like allowing `kms:*` to all principals.

### 2. What are KMS grants and when would you use them instead of key policies?

**Answer:**
A KMS grant is a mechanism that allows you to delegate a subset of key usage permissions to a principal (typically an AWS service or another AWS account) **without modifying the key policy**. Grants are temporary, programmatic, and auditable.

**When to use grants:**
- **AWS services acting on your behalf**: Services like AWS Secrets Manager, EBS, or RDS create grants when they need to use a KMS key to perform operations on your behalf. This avoids modifying the key policy every time a new resource is created.
- **Temporary delegated access**: When you want to grant a Lambda function or EC2 instance temporary access to use a key without permanently modifying the key policy.
- **Fine-grained operations**: Grants can restrict access to specific operations (e.g., only `Decrypt`, not `Encrypt`), specific encryption contexts, or specific principals.

**Key properties:**
- Created with `CreateGrant` API
- Can include a **retiring principal** who can revoke the grant
- Grants are **eventually consistent** — there's a small delay before they take effect (use `RetireGrant` or `RevokeGrant` for immediate removal)
- Grants support **grant tokens** to handle the eventual consistency window
- Grants appear in CloudTrail logs

**Example use case**: When you attach an EBS volume encrypted with a customer managed KMS key to an EC2 instance, AWS creates a grant on the key for the EC2 service to decrypt the volume. When the volume is detached, the grant is retired.

### 3. What is an encryption context in KMS and why is it important?

**Answer:**
An encryption context is a set of optional, non-secret key-value pairs that provide additional authenticated data (AAD) for cryptographic operations. It is used with symmetric KMS keys.

**How it works:**
- When you encrypt data with an encryption context, KMS binds that context to the ciphertext using authenticated encryption (AES-GCM).
- To decrypt, you must provide the **exact same encryption context**. If it doesn't match, decryption fails — even with a valid KMS key.
- The encryption context is **not encrypted** — it appears in plaintext in CloudTrail logs.

**Why it's important:**

1. **Additional security layer**: Prevents decryption of data in the wrong context. For example, encrypting a file for "project=A" means it cannot be decrypted in the context of "project=B."

2. **Audit and compliance**: The encryption context appears in CloudTrail, allowing you to trace exactly what data was encrypted/decrypted and in what context.

3. **Key policy conditions**: You can write key policy conditions that require a specific encryption context, limiting key usage to only the intended application or resource.

```json
"Condition": {
  "StringEquals": {
    "kms:EncryptionContext:Application": "PaymentService"
  }
}
```

4. **Defense against confused deputy attacks**: Binding context to ciphertext ensures that even if an attacker obtains the ciphertext, they cannot decrypt it without the matching context.

### 4. How does KMS integrate with S3 Server-Side Encryption? What are the differences between SSE-S3, SSE-KMS, and SSE-C?

**Answer:**

| Feature | SSE-S3 (AES-256) | SSE-KMS | SSE-C |
|---|---|---|---|
| Key management | AWS manages entirely | AWS KMS (AWS or Customer Managed) | Customer provides key per request |
| Key visibility | None | Full audit in CloudTrail | Customer-side only |
| Cost | No additional cost | KMS API call charges | No KMS cost |
| Key rotation | Automatic by AWS | Configurable | Customer responsibility |
| Cross-account access | Not applicable | Supported via key policy | Not applicable |
| Envelope encryption | Yes (internal) | Yes (explicit) | No |

**SSE-KMS specifics:**
- Each object is encrypted with a unique DEK, which is itself encrypted with the KMS key.
- Every `GET` and `PUT` of an encrypted object generates a KMS API call, which can hit **KMS request quotas** (5,500–30,000 requests/second depending on region).
- You can use `aws/s3` (AWS managed) or a customer managed KMS key.
- With customer managed keys, you can enforce bucket-level encryption using bucket policies that deny `s3:PutObject` unless the request includes `x-amz-server-side-encryption: aws:kms`.
- **S3 Bucket Keys**: A feature that reduces KMS API calls by generating a short-lived bucket-level key that encrypts individual object DEKs locally, reducing KMS calls by up to 99%.

**S3 Bucket Key consideration**: While it significantly reduces cost and API call volume, it changes the granularity of CloudTrail logging — you see one KMS call per bucket key generation rather than per object.

### 5. What is the KMS request quota and how do you handle throttling?

**Answer:**
KMS enforces **request quotas** (formerly called limits) on the number of cryptographic API calls per second, per account, per region. These quotas are shared across all KMS keys in the account.

**Default quotas (approximate, varies by region and API):**
- Symmetric key operations (Encrypt, Decrypt, GenerateDataKey, etc.): 5,500–30,000 requests/second
- Asymmetric key operations: Lower (typically 500 requests/second for RSA operations)
- GenerateDataKeyWithoutPlaintext: Shares quota with other symmetric operations

**Handling throttling:**

1. **Use S3 Bucket Keys**: Reduces KMS calls for S3 workloads by up to 99%.

2. **Data key caching**: Use the AWS Encryption SDK's data key caching feature. Cache DEKs locally (in memory) for a configurable TTL and reuse them for multiple encrypt/decrypt operations instead of calling KMS for each one. Balance security (shorter TTL = more secure) vs. performance (longer TTL = fewer KMS calls).

3. **Request quota increases**: Submit a Service Quota increase request through the AWS console or Support.

4. **Exponential backoff with jitter**: Implement retry logic with exponential backoff when receiving `ThrottlingException`.

5. **Distribute across regions**: For multi-region workloads, use multi-region keys or per-region keys to distribute API calls.

6. **Review architecture**: Avoid calling KMS for every individual record in a high-throughput data pipeline; instead, use envelope encryption and cache DEKs.

7. **Monitor with CloudWatch**: Track `KMS.RequestCount` and set alarms before hitting quota limits.

## Hard

### 1. Explain KMS multi-region keys, their use cases, architecture, and security considerations.

**Answer:**
Multi-Region Keys (MRKs) are a set of KMS keys in different AWS Regions that have the **same key material and key ID prefix** but different ARNs. They enable you to encrypt data in one Region and decrypt it in another without re-encrypting the data or making cross-region KMS calls.

**Architecture:**
- A **primary key** is created in one Region.
- **Replica keys** are created in other Regions from the primary key.
- All keys in the set share the same key material (same underlying cryptographic key) and the same key ID suffix (e.g., `mrk-1234abcd...`).
- Each replica has its own key policy, tags, and enabled/disabled state.
- Key material is synchronized by KMS using a secure, hardware-protected process.

**Use cases:**

1. **Active-active multi-region architectures**: Applications in `us-east-1` and `eu-west-1` can encrypt and decrypt data locally without cross-region API calls, reducing latency.

2. **Disaster recovery**: If the primary Region becomes unavailable, applications in DR Regions can decrypt data using the replica key.

3. **Global data lakes**: Data encrypted in one Region can be read and processed in another Region without re-encryption.

4. **Cross-region database replication**: Aurora Global Database with customer managed KMS keys can use MRKs so the replica Region can decrypt data without cross-region KMS calls.

**Security considerations:**

- **Shared key material is a deliberate trade-off**: The same key material in multiple Regions increases the attack surface. Compromising the key material in one Region compromises all Regions.
- **Independent key policies**: Each replica has its own key policy, allowing Region-specific access control. A principal with access in one Region does not automatically have access in another.
- **Rotation**: When you rotate an MRK, the rotation applies to the primary key and is synchronized to all replicas.
- **Deletion**: You must delete all replicas before deleting the primary key. Each deletion has a 7–30 day waiting period.
- **CloudTrail**: Decryption in each Region generates its own CloudTrail event in that Region — ensure log aggregation for complete audit trails.
- **Not a replacement for cross-region replication policies**: Access control must be independently configured per Region.

**When NOT to use MRKs:**
- When your compliance requirements mandate that key material never exists in certain Regions.
- When the shared key material risk outweighs the operational benefit.

### 2. How does KMS implement the security model to ensure keys never leave the service unencrypted? What is the role of HSMs?

**Answer:**
KMS is built on a foundation of **FIPS 140-2 validated Hardware Security Modules (HSMs)**, specifically AWS-designed HSMs that form the KMS fleet. The security model operates at multiple layers:

**HSM Architecture:**
- KMS key material is generated inside HSMs and is **never exported in plaintext** from the HSM boundary.
- Each KMS key's plaintext key material exists only in HSM volatile memory during cryptographic operations.
- HSMs are physically tamper-resistant — any attempt to open or tamper with the device causes automatic key material erasure.
- AWS KMS HSMs are validated to **FIPS 140-2 Level 2** overall, with Level 3 for the cryptographic module.

**Key material protection layers:**

1. **Domain keys**: Each KMS HSM cluster is protected by domain keys that are themselves distributed across multiple HSMs. No single HSM holds all domain key shares (Shamir's Secret Sharing scheme).

2. **HSM backing keys (HBKs)**: The actual KMS key material is encrypted under HBKs that exist only within the HSM. When a KMS key is "at rest" in the KMS database, it is stored as ciphertext encrypted by the HSM.

3. **Operator key cards**: Administrative operations on the HSM fleet require physical key cards held by multiple AWS employees — no single person can access the HSMs alone (two-person integrity).

**Cryptographic operations flow:**
1. Client calls `kms:Decrypt` with ciphertext.
2. KMS authenticates and authorizes the request (IAM + key policy).
3. KMS retrieves the encrypted key material from its database.
4. The encrypted key material is loaded into the HSM, decrypted within the HSM boundary.
5. The HSM performs the decryption operation using the plaintext key (which never leaves the HSM).
6. The plaintext result is returned to the client over a TLS-encrypted channel.
7. The plaintext key material is zeroed in HSM memory.

**CloudHSM vs KMS HSMs:**
- KMS uses AWS-managed HSMs that you cannot access directly.
- AWS CloudHSM gives you dedicated HSMs with direct access, but you manage all key material and operations yourself.
- KMS Custom Key Stores (backed by CloudHSM or external key stores) allow you to use KMS APIs while keeping key material in your own HSM cluster.

**KMS Custom Key Stores:**
- **CloudHSM-backed**: Key material lives in your CloudHSM cluster. You maintain control but also responsibility.
- **External Key Store (XKS)**: Key material lives in an on-premises HSM or external key manager. KMS proxies cryptographic operations to your external key manager. Useful for compliance requirements mandating on-premises key storage.
